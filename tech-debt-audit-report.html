The AUSTA healthcare super-app monorepo contains 7 backend NestJS microservices, a Next.js web app, a React Native mobile app, and a shared design system. This audit examined code quality, architecture, test coverage, dependency health, infrastructure, and documentation across ~550 source files, ~135 backend unit tests, 65 E2E/integration test cases, and ~692 frontend tests.

The overall health is moderate, with mature CI/CD, solid E2E test infrastructure (14 spec files across all services), and comprehensive frontend testing, but significant gaps in unit test coverage for key services, dependency hygiene, and architectural boundaries. Four critical items require immediate attention to prevent build instability and repository bloat.

Detailed Findings by Category

Code Debt

34 files exceed the 500-line limit — The worst offender is appointments.service.ts at 1,004 lines with 15 functions. Other large files include the GoogleFit adapter (787 lines), redis.service.ts (634 lines), and validation.ts (624 lines). Each should be split into focused modules by responsibility.
838 uses of any across 163 files — Concentrated in rules.service.ts (22 uses), redis.service.ts (16), claims.service.ts (16), and notifications.service.ts (14). While many are in test files, production services lose type safety at integration boundaries where it matters most.
Nested directory duplication: src/web/web/ — While the nested structure is intentional (Turbo workspace pattern), the inner web/web/ directory contains its own node_modules/, .next/ build artifacts, duplicate middleware.ts, and duplicate config files. Build artifacts and duplicated middleware create ambiguity about the source of truth.
37 console.log statements and 12 TODO markers — Most console.log calls are in seed files and Storybook stories (acceptable). Two security-relevant TODOs stand out: SSL pinning with placeholder certificates and mobile app integrity checks not yet implemented.

Architecture Debt

Cross-service imports violate bounded contexts — care-service directly imports auth decorators (@CurrentUser, @Roles) from auth-service/src/auth/decorators/ via relative path traversal (../../../auth-service). This creates tight coupling between services and would prevent independent deployment.
Shared package is a monolith — The @austa/shared package contains 18 modules and 98 files spanning audit, consent, privacy (GDPR/HIPAA), encryption, Kafka, Redis, logging, tracing, and more. Domain logic (consent, privacy) is mixed with infrastructure utilities (Redis, Kafka), making the shared package a bottleneck for every service change.
No service API facade interfaces — Services communicate through shared utilities and Kafka events, but there are no explicit public API interfaces defining service contracts. Any internal refactor risks breaking consumers silently.

Test Debt

health-service: 15.28% line coverage, 9.72% branch coverage — Only 14 tests for 51 source files. Devices controller and service at 0% coverage. FHIR and wearables integrations untested. Many test files are skeleton-only (should be defined assertions). This is the highest-risk service given it handles patient health data.
E2E tests: 65 test cases across 14 spec files + 2 integration specs — [CORRECTED from initial audit] Backend E2E testing is substantive, using NestJS TestingModule with supertest. Coverage spans all 7 services: auth (login, register, refresh, users CRUD), appointments (CRUD + permissions), telemedicine (sessions + auth), health metrics (creation, updates, invalid input), devices (connect, retrieve), notifications (send, validate, journey templates), claims (lifecycle + Kafka events), plans (CRUD + access control + coverage + benefits), achievements, and gamification events. Additionally, claims.integration.spec.ts (7 tests) and payment.integration.spec.ts (5 tests) cover cross-service lifecycle flows. Each service has a jest-e2e.json config. Remaining gap: No browser-level E2E tests (Playwright/Cypress) for frontend user flows.
care-service: 47.74% line coverage — Appointments controller (6 functions) and service (15 functions, 189 lines) both show 0% in coverage baseline despite a 319-line spec file existing. Possible instrumentation issue, or tests are not running against the coverage target.
Branch coverage: 9–40% across all services — Error paths, conditional logic, and edge cases are systematically untested. Even well-covered services like gamification-engine (85% line) only reach 39% branch coverage.

Dependency Debt

42 MB binary and 5.7 MB database files in Git — next-swc-darwin-arm64-15.5.12.tgz (42 MB, platform-specific), ruvector.db (1.6 MB), and vectors.db (4.1 MB) are committed. This bloats every clone and is not cross-platform safe. These should be in .gitignore and removed from history via git filter-repo.
89 dependency resolutions in root package.json — A mix of security patches (follow-redirects, minimist, glob), compatibility hacks (50+ Babel transforms), and version pins. The React ecosystem shows a direct conflict: root resolves react@18.3.1 while the web workspace declares react@19.2.4. NestJS versions drift between root (11.1.16) and backend (11.0.16).
React Native ecosystem 3+ versions behind — Navigation (v6 vs v7 available), Reanimated (3.4 vs 3.17), Gesture Handler (2.12 vs 2.25). Security and performance patches are accumulating.

Infrastructure Debt

Duplicate infrastructure directories — Both /infra/ (only nginx config) and /infrastructure/ (Terraform, Kubernetes, monitoring, Docker Compose) exist. The /infra/ directory appears abandoned.
5 Docker Compose files scattered — docker-compose.yml, docker-compose.staging.yml, and docker-compose.scale.yml in /src/backend/, docker-compose.production.yml in /infrastructure/, and docker-compose.local.yml in /src/backend/gamification-engine/. No documented guidance on which to use when.
Strong CI/CD (12 workflows) but no documented procedures — Deployment, rollback, and incident response runbooks are absent. The production deployment workflow is 16 KB of YAML but has no companion documentation.

Documentation Debt

No operational runbooks — No documented procedures for emergency incident response, infrastructure scaling, database migrations, deployment rollback, or monitoring alert triage. All 7 backend services have README files (good), but operational knowledge is tribal.
No dependency management strategy doc — Despite a detailed Renovate config (325 lines), there is no developer-facing document explaining the resolution strategy, why 89 packages are pinned, or how to handle version conflicts.
Strong foundations — The .env.example (38 KB, 50+ variables with [REQUIRED] / [SECRET] markers), 6 Architecture Decision Records, comprehensive main README (767 lines), and per-service README files provide a solid documentation baseline.

Phased Remediation Plan

1
Week 1–2: Critical Fixes — Stop the bleeding
  • Remove binaries from Git — Add *.tgz, *.db, .DS_Store to .gitignore. Run git filter-repo to purge from history. Saves 48 MB per clone. Effort: 1 day.
  • Fix React version conflict — Align root resolution and web workspace to the same React version (18 or 19). Test build across web and mobile. Effort: 1–2 days.
  • Replace SSL pinning placeholder certs — Address TODO(ops) in ssl-pinning.ts and plans.e2e-spec.ts with real certificate SPKI hashes. Effort: 1 day.
  • Delete /infra/ directory — Consolidate into /infrastructure/. Effort: 30 minutes.
  • Extract auth decorators to shared — Move @CurrentUser and @Roles from auth-service internals to the shared package or a dedicated @austa/auth-contracts package. Fix care-service imports. Effort: 2 days.
2
Week 3–6: Test Coverage Sprint — Build the safety net
  • health-service: 15% → 70% — Add ~40 tests covering devices, FHIR integration, wearables adapters, and health record CRUD. Focus on branch coverage for medical data validation. Effort: 2 weeks.
  • care-service: 48% → 70% — Fix coverage instrumentation for appointments. Add tests for medication management, treatment plans, telemedicine session handling. Effort: 1 week.
  • Expand E2E test depth — Backend E2E is solid (65 cases). Add browser-level E2E with Playwright for 5 critical frontend flows: login, appointment booking, health metric entry, claims submission, gamification reward redemption. Effort: 2 weeks.
  • Raise branch coverage to 50%+ — Add error-path and edge-case tests across all services. Focus on shared package (currently 62% branch). Effort: ongoing alongside feature work.
3
Week 7–10: Architecture & Dependency Cleanup — Reduce coupling
  • Split shared package — Extract domain logic (consent, privacy, audit) into @austa/compliance. Keep only infrastructure utils (database, redis, kafka, logging, tracing) in shared. Effort: 2 weeks.
  • Define service API contracts — Create TypeScript interface packages (@austa/care-contracts, @austa/health-contracts) defining public APIs. Services import contracts, not implementations. Effort: 1 week.
  • Reduce dependency resolutions — Audit all 89 resolutions. Remove those already satisfied by updated dependencies. Target reducing to <30 resolutions. Effort: 1 week.
  • Update React Native ecosystem — Upgrade navigation to v7, reanimated to 3.17, gesture-handler to 2.25. Test on both iOS and Android. Effort: 1 week.
  • Break down large files — Split appointments.service.ts (1,004 lines) into scheduling, validation, and notification sub-services. Split GoogleFit adapter (787 lines) by data type. Effort: 1 week.
4
Week 11–14: Polish & Prevention — Sustain quality
  • Write operational runbooks — Deployment, rollback, incident response, database migration, monitoring alert triage. Store in /docs/runbooks/. Effort: 1 week.
  • Consolidate Docker Compose files — Move all to /infrastructure/docker/ with clear naming (dev, staging, production, scale). Document in README. Effort: 2 days.
  • Eliminate any types in production code — Start with highest-impact files: rules.service.ts (22), redis.service.ts (16), claims.service.ts (16). Add ESLint rule @typescript-eslint/no-explicit-any: warn. Effort: 1 week.
  • Add pre-commit quality gates — Enforce file size limits (500 lines), minimum coverage thresholds per service, and any type budgets via Husky hooks. Effort: 2 days.
  • Write dependency management guide — Document resolution strategy, Renovate config rationale, and upgrade procedures. Effort: 1 day.
  • Clean up remaining items — Remove .bak files, legacy /docs/original documentation/, console.log in components, and resolve 12 TODO markers. Effort: 2 days.
